# v0.2.2 Roadmap: Hardware-Close Decode Path

This document turns the next-step direction into a concrete implementation
plan for the repository.  The goal is to move one path from
"control-plane prototype" to "real decode math path" while preserving the
repo's benchmarking honesty and close-to-hardware focus.

## Goal

Deliver one real, narrow decode fast path that:

- runs inside the persistent worker model already used by the runtime
- performs actual single-token decode attention math
- uses a hardware-shaped KV memory layout
- has correctness tests against a simple reference path
- reports performance separately from the control-plane-only benchmarks

This is intentionally not a general inference framework.  It is one
well-measured fast path.

## Fixed-Scope Configuration

For `v0.2.2`, keep the first real path intentionally narrow:

- single-token decode only
- one fixed head dimension, recommended `128`
- one KV block layout
- one precision mode, recommended `fp16` or `bf16`
- one attention kernel path
- one GPU architecture tuning target first, then validate on others

This narrow scope is the main protection against accidentally building a
slow abstraction layer.

## Success Criteria

`v0.2.2` is successful when all of the following are true:

1. `cu/worker.cu` can execute a real decode step instead of a mock
   completion-only stub.
2. The real path is benchmarked separately from the control-path-only
   path.
3. There is at least one CUDA correctness test comparing the kernel
   output against a small reference implementation.
4. README and benchmark output clearly distinguish control-plane
   measurements from real decode math measurements.

## File-by-File Breakdown

### New files

#### `include/kv_layout.h`

Purpose:
- define the concrete KV-cache block layout used by the real decode path

Contents:
- fixed constants such as:
  - `KV_HEAD_DIM`
- `KV_TOKENS_PER_BLOCK`
  - `KV_VEC_BYTES`
  - `KV_ALIGNMENT`
- packed or alignment-sensitive structs for K/V tiles
- helpers for:
  - block base address computation
  - token offset computation inside a block
  - K/V plane addressing

Add static assertions for:
- alignment
- block-size divisibility
- vector load width compatibility
- expected block byte size

Suggested shape:

```c
typedef struct {
    uint32_t token_count;
    uint32_t head_dim;
    uint32_t k_stride_bytes;
    uint32_t v_stride_bytes;
    uint32_t block_bytes;
} KvLayoutDesc;
```

Keep this descriptor compile-time-oriented, not heap-configurable.

#### `include/attention_decode.h`

Purpose:
- define the real decode kernel interface and fixed-path constants

Contents:
- kernel launch helper prototypes
- per-kernel constants:
  - tile sizes
  - stages in prefetch pipeline
  - warps per block
- small POD structs for decode arguments

Suggested types:

```c
typedef struct {
    const void *q_ptr;
    const void *k_base;
    const void *v_base;
    void       *o_ptr;
    uint32_t    seq_len;
    uint32_t    head_dim;
    uint32_t    kv_block_base;
    uint32_t    kv_block_count;
} DecodeStepArgs;
```

If the first implementation uses resident state instead of explicit
pointer passing, keep this struct small and only carry what the kernel
actually needs.

#### `cu/attention_decode.cu`

Purpose:
- own the actual single-token decode math path

Responsibilities:
- load query tile
- iterate over KV tiles
- prefetch K/V into shared memory
- apply RoPE if enabled for the chosen fixed path
- compute QK dot products
- apply causal masking
- perform stable softmax
- accumulate V
- optionally emit logits or final hidden/output vector

Suggested internal structure:
- `load_q_fragment(...)`
- `prefetch_kv_tile(...)`
- `compute_qk_tile(...)`
- `softmax_update(...)`
- `accumulate_v_tile(...)`
- `store_decode_output(...)`

Do not make this generic for many head dims in the first version.
Specialize for one shape and use constants.

#### `bench/bench_decode_microkernel.cu`

Purpose:
- measure only the real decode kernel path

Report:
- tokens/s
- ns/token
- achieved KV bandwidth estimate
- effective bytes read per token
- launch geometry
- shared memory bytes
- register pressure if surfaced manually

CLI suggestions:
- `--seq-len`
- `--iters`
- `--warmup`
- `--dev`
- `--precision`

This benchmark should not route through rollout/pipeline logic.

#### `bench/bench_kv_layout.cu`

Purpose:
- isolate KV layout and memory movement costs

Measure:
- linear vs blocked access patterns
- decode-time K/V tile gather cost
- shared-memory stage throughput
- alignment sensitivity

This benchmark is important because memory layout decisions will likely
matter more than scheduler ideas at this stage.

#### `docs/decode_microkernel.md`

Purpose:
- explain the fixed decode path and what it is proving

Document:
- chosen head dimension
- chosen tile shape
- shared-memory usage
- register pressure tradeoffs
- known limitations
- what is still stubbed after `v0.2.2`

### Existing files to modify

#### `cu/worker.cu`

Current state:
- `process_descriptor()` is a stub that validates queue flow

`v0.2.2` changes:
- replace the stub body with a call into `attention_decode.cu`
- keep completion push, step counting, and queue orchestration here
- keep `worker.cu` focused on:
  - descriptor intake
  - worker loop
  - completion emission
  - shutdown behavior

Recommended boundary:
- orchestration in `worker.cu`
- math in `attention_decode.cu`

#### `cu/prefetch.cu`

Current state:
- prototype-level support

`v0.2.2` changes:
- turn it into a real helper for K/V staged loads
- define one preferred staging policy:
  - 2-stage or 3-stage software pipeline
- tune for:
  - global-memory coalescing
  - shared-memory bank behavior
  - occupancy impact

Avoid adding many variants until the first one is measured.

#### `include/descriptor.h`

Current state:
- descriptor is sized for the control-path prototype

`v0.2.2` changes:
- extend only if the real decode path needs more information

Candidate additions:
- `seq_len`
- `head_dim`
- `q_offset`
- `output_offset`
- `kv_base_block`
- flags for:
  - RoPE enabled
  - causal-only fast path

Do not add speculative fields.  Every new field should be justified by
the actual kernel inputs.

#### `include/sample.h` and `cu/sample.cu`

Current state:
- already provides GPU-side sampling primitives

`v0.2.2` guidance:
- keep sampling mostly stable
- use it only after the real decode path emits real logits or token
  candidates
- do not expand sampling complexity until the decode kernel is correct

#### `bench/bench_pipeline.cu`

Current state:
- measures the broader runtime flow

`v0.2.2` changes:
- keep this benchmark
- do not reinterpret it as "real model throughput"
- if the real math path is wired into it, add an explicit mode label such
  as:
  - `control_stub`
  - `real_decode_fixed128`

#### `test/test_bench.cu`

Current state:
- mixed unit and CUDA test coverage

`v0.2.2` changes:
- add a small CUDA correctness test for the real decode kernel
- compare against a tiny CPU reference implementation for:
  - dot product
  - masking
  - softmax
  - V accumulation

Suggested test cases:
- sequence length `1`
- sequence length `16`
- sequence length `64`
- deterministic query/K/V tensors

### Optional new test file

#### `test/test_decode_reference.cu`

Purpose:
- keep the real-kernel correctness tests isolated from the broader
  runtime suite

This is useful if the CUDA math tests grow enough to deserve their own
binary or `make` target.

## Descriptor Change Proposal

If the real path needs a slightly richer descriptor, prefer something
like this:

```c
typedef struct {
    uint64_t seq_id;
    uint32_t kv_block_base;
    uint16_t kv_block_count;
    uint16_t seq_len;
    uint32_t q_offset;
    uint32_t output_offset;
    uint16_t head_dim;
    uint8_t  attention_flags;
    uint8_t  reserved;
    uint64_t reward_cookie;
} DescriptorV2;
```

This is only a proposal.  Keep the actual descriptor smaller if some
fields can be derived from resident rollout state.

## Benchmark Plan

`v0.2.2` should keep two benchmark classes clearly separate.

### Control-path benchmarks

Existing style:
- ring overhead
- pipeline flow
- scheduler/control tax

These answer:
- how much CPU/GPU coordination costs
- whether hot-path invariants hold
- whether queueing and backpressure behave well

### Real decode benchmarks

New style:
- decode microkernel only
- KV layout bandwidth
- optional end-to-end persistent-worker decode with real math enabled

These answer:
- how close the math path is to hardware limits
- whether the chosen memory layout is efficient
- where the real decode latency is being spent

## Non-Goals for `v0.2.2`

Do not try to do these in the same release:

- many head dimensions
- multi-GPU support
- generalized model graph execution
- broad runtime configurability
- production-grade attention kernel coverage for all shapes

Those are later phases.  `v0.2.2` should prove one real path first.

## Recommended Starting Order

1. Add `include/kv_layout.h`.
2. Add `include/attention_decode.h`.
3. Create `cu/attention_decode.cu` with a fixed-shape kernel.
4. Add `bench/bench_decode_microkernel.cu`.
5. Add correctness checks in `test/test_bench.cu` or a dedicated decode
   test file.
6. Only after the microbenchmark looks credible, wire it into
   `cu/worker.cu`.

That ordering keeps the work measurable and keeps the repo honest about
what is implemented versus what is merely planned.
