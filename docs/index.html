GB300 RL Runtime · C/CUDA · Hardware-Shaped Inference

Persistent workers, lock-free rings, batched descriptor windows, and a growing decode pipeline.

gb300-rl-runtime is a close-to-metal reference runtime for reinforcement-learning inference. It focuses on the control-path and data-path mechanics that matter when the target is low-latency decode: coherent rings, arena-managed KV, GPU-resident workers, explicit queue windows, and increasingly hardware-shaped decode kernels.

Why This Repo Exists

This project is less about model quality and more about runtime shape. It explores what an RL inference fast-path looks like when the interesting problems are ring semantics, device-visible scheduling, pre-allocated KV, and decode flow control rather than framework breadth.

Hot Path Discipline

The repo treats the per-token loop as a systems problem. Rings, arena allocation, completion flow, and persistent workers are all designed around keeping the decode hot path predictable.

System Snapshot

The runtime is built from a small set of explicit primitives. The control plane lives in coherent or pinned memory. The device side polls, prefetches, decodes, and completes. Recent work has added decode queue windows and grouped descriptor submission on the host side.

Host Control Plane

  • CommandRing publishes decode descriptors from CPU to GPU.
  • RolloutPipeline tracks rollout movement across free, prefill, decode, reward, trajectory, and done queues.
  • PipelineSnapshot exposes queue occupancy, credit headroom, and suggested batch windows.
  • DecodeDispatchBatch groups descriptor windows and commits them to the ring in one shot.

GPU Data Path

  • decode_worker runs as a persistent kernel, one warp per SM today.
  • model_state produces a deterministic residual-style hidden-state update.
  • query_producer projects hidden state into explicit decode query buffers.
  • attention_decode_step_fixed128 executes real QK / softmax / V math on one fixed path.

Implementation Status

The project draws a sharp line between what is real, what is partial, and what is still scaffolding. That line matters because the repo is intended to be trusted as a systems prototype, not just a benchmark shell.

Area Status What It Actually Means
Command and Completion Rings Real SPSC semantics are corrected and tested, including wraparound, consumer progress visibility, and no permanent-full failure mode.
KV Arena and Host Runtime Plumbing Real Pre-allocated arena, runtime init/shutdown, persistent worker launch, and explicit query/output buffer plumbing are present.
Fixed128 Decode Path Partial Real attention math exists for one fixed path. The kernel is warp-cooperative and the score pass now uses 8-lane row groups, but it remains single-warp and single-block.
Model-State and Query Stages Partial The runtime now has explicit model_state -> query_producer -> decode stages, but they still use deterministic scaffolding rather than trained model activations.
Pipeline Windows and Decode Admission Real Decode queue occupancy, stage credits, batch windows, batch-driven decode admission, and grouped descriptor submission are all wired together in the benchmark path.
Reward / Full RL Semantics Stub Reward scoring and broader model semantics remain mock or narrow; the repo is still primarily a runtime/control-path prototype.

Current Priorities

  • Move from host-visible batch contracts toward GPU-visible decode batch semantics.
  • Broaden the fixed128 decode path from one staged KV block to multiple blocks per step.
  • Replace deterministic model-state scaffolding with a more realistic activation source.
  • Keep strengthening tests and docs so low-level changes stay honest and reviewable.

What This Is Not

This repo is not pretending to be a production LLM serving framework today. It is narrower and, in some ways, more useful because of that.

Not a Full Serving Stack

It does not yet compete with broad systems like vLLM, TensorRT-LLM, or SGLang on feature completeness, model support, or operational polish.

A Reference Fast Path

It is a detailed reference for how a low-level RL inference runtime can be decomposed into queues, credit windows, descriptor windows, resident workers, and a gradually more real decode path.
